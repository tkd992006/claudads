광고 노출 알고리즘

데스크탑이 /api/ads/serve 를 때리면, 서버는 selectAdForUser(userId) 로 후보를 추리고 가중치 랜덤으로 하나를 골라 임프레션 레코드를 만든 뒤 응답합니다. 전체 로직은 web/lib/ads.ts:17-62 한 함수에 들어있습니다.

왜 "표시할 광고가 없습니다" 가 뜨는가

한 번 사용자에게 노출된 광고는 24시간 동안 동일 사용자에게 다시 안 뜹니다 (primary 티어 한정). 후보 쿼리의 impressions: { none: { userId, startedAt: { gte: 24h전 } } } 조건이 이 dedup 을 강제하기 때문입니다. 게다가 완료 여부와 무관 하게 startedAt 만 보기 때문에, 영상을 끝까지 안 봐도 (=토큰 적립 X) 그 광고는 24시간 잠깁니다.

단, primary 가 비면 fallback 티어 가 24h dedup 을 풀고 같은 풀을 LRU 기반으로 다시 회전시킵니다. 따라서 "광고 없음" 빈 응답은 사실상 전체 풀이 비었거나 예산/스케줄이 모두 막혔을 때만 발생합니다.

1 전체 흐름 한 줄 요약

A Primary 후보 1차 필터 (Prisma where, 24h dedup 포함)
B 2차 필터 (JS 루프: 예산/일캡)
C 잔여 예산 × 지터 가중치 랜덤 선택
D Primary 가 비면 Fallback 티어 (24h dedup 해제, LRU 가중치)
E Impression 행 생성, 영상 signed URL 반환 (isFallback 플래그 포함)

아래 4가지를 모두 만족해야 후보가 됩니다.

DB ① 승인 상태

3 2차 필터 — JS 루프

DB 에서 빠진 후보들을 메모리에서 한 번 더 거릅니다.

JS ④ 예산 소진 여부
spentCents >= budgetCapCents 면 제외. impression 한 번당 ceil(cpmCents / 1000) 만큼 차감되어 쌓이고, 다 쓰면 status 가 EXHAUSTED 로 바뀌어서 사실상 ①에서도 막힙니다.

4 선택 — 가중치 랜덤

eligible 리스트가 비면 null 리턴 → 클라이언트에 빈 응답. 비어 있지 않으면 광고별로 가중치를 매겨 랜덤 추첨: 

weight_i = max(1, budgetCapCents_i - spentCents_i) * (0.5 + Math.random())

잔여 예산이 많을수록 뽑힐 확률이 높고, 0.5 ~ 1.5 의 곱셈 지터로 결정론적 편향을 약하게 흔들어 줍니다. 누적 분포에서 Math.random() * total 위치를 잘라서 하나를 고르는 weighted-roulette 방식이에요.

web/lib/ads.ts:50-61

5 Fallback 티어 (Primary 가 빌 때)

오늘 적격 광고를 다 본 사용자에게도 광고는 계속 노출되어야 합니다. Primary 가 null 을 리턴하면 selectFallbackAdForUser(userId) 가 호출되어 같은 1·2차 필터를 다시 돌리되 24h dedup 만 풉니다. 그 다음 LRU 기반 회전 가중치로 weighted random.

JS 신선도(freshness) × 잔여예산 × 지터

이 유저에게 마지막으로 보여준 시각을 광고별로 계산 (impression.groupBy(by:adId, _max: startedAt)), 경과 시간을 0~24h 로 클램프해서 가중치로 씁니다. 

minsSinceLast = clamp((now - lastShownAt) / 60s, 0, 1440)
freshness     = minsSinceLast + 1        // 1 ~ 1441
weight_i      = freshness * max(1, remaining_budget) * jitter(0.7~1.3)
  • 방금 본 광고 → freshness ≈ 1 (확률 거의 0)
  • 1시간 전 → 61
  • 12시간 전 → 721
  • 24h 이상 / 한 번도 못 본 광고 → 1441 (최대)
web/lib/ads.ts · selectFallbackAdForUser

결과 광고는 같은 recordImpressionStart 경로로 들어가서 impression 행이 만들어지고, 토큰 보상도 동일하게 지급됩니다. 응답엔 isFallback: true 플래그가 붙어요 (분석/UI 용도).

광고주 과금 주의

Fallback 도 recordImpressionComplete 에서 동일하게 spentCents += ceil(cpmCents/1000) 차감합니다. 즉 fallback 으로 재노출된 impression 도 광고주에게 과금됩니다. 한 유저가 하루에 같은 광고를 여러 번 카운트로 보게 되면 reach 가 부풀려질 수 있으니, 필요하면 fallback impression 은 0원 (또는 할인 CPM) 으로 따로 처리하는 정책을 추가하세요. 지금은 정책 없음 = 같은 가격.

6 Impression 생성 + 응답

뽑힌 광고에 대해 recordImpressionStart 가 Impression { adId, userId, deviceId, ipHash } 행을 만듭니다. 이 시점부터 ③의 24h dedup 이 발동합니다. 그리고 R2 signed URL 로 영상 주소를 만들어 응답. 

{
  impressionId,
  isFallback,             // true 면 fallback 티어로 골라진 광고
  ad: { id, title, videoUrl, durationSec,
        cta: ctaLabel && ctaUrl ? { label, url } : null }
}

고스트 임프레션 주의

이전엔 AdOverlay 가 busy 플래그를 따라 마운트/언마운트되었는데, Claude 의 툴 호출이 보통 영상 길이보다 짧기 때문에 영상이 끝나기 전에 overlay 가 사라지는 일이 잦았습니다. 그 경우 completedAt 이 안 박힌 "started-but-never-completed" impression 행이 DB 에 남고, ③의 dedup 이 그걸 보고 그 광고를 24시간 동안 잠가버립니다. 토큰은 안 쌓이는데 광고는 또 안 뜨는 상태.

오버레이 라이프사이클 버그는 이미 고쳤지만 (App.tsx / AdOverlay.tsx), 기존에 쌓인 고스트 임프레션은 테이블에서 직접 지워야 풀립니다: 

-- 미완료 임프레션만 정리 (Prisma 콘솔이나 psql)
DELETE FROM "Impression"
WHERE "completedAt" IS NULL
  AND "startedAt" > now() - interval '24 hours';
설계 의도

24h-per-ad-per-user dedup 은 광고주 입장에선 같은 사람한테 하루에 한 번까지만 카운트되어 reach 가 부풀려지지 않게, 시청자 입장에선 같은 광고를 반복해서 보는 피로를 막아주는 장치입니다. 다만 광고 풀이 작을 때(=초기) 빠르게 풀이 비어 보이는 부작용이 있어요. 라이브에서 풀을 키우거나, 시청자 cooldown을 좁히는 (예: 6h) 식의 튜닝이 필요할 수 있습니다.
